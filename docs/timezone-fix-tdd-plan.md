# План внедрения исправлений timezone-проблемы через TDD

## Цель

Подготовить безопасное поэтапное исправление проблемы с датами без изменений на сервере:

1. минимально рискованный hotfix для уже выпущенного приложения;
2. полный client-only fix на уровне сериализации, десериализации и sync-логики;
3. обязательный набор тестов, который должен защитить от регрессий.

Документ описывает только план. Исправления в коде на этом этапе не выполняются.

## Принципы выполнения

- Использовать TDD на всех этапах: `Red -> Green -> Refactor`.
- Сначала писать узкие целевые тесты на текущее некорректное поведение, затем вносить минимальное исправление.
- Для hotfix не затрагивать лишние части sync-логики, пока они не покрыты тестами.
- После каждого этапа запускать форматирование и только самые целевые тесты, затем расширять прогон.
- Для дат в клиенте считать источником истины абсолютный момент времени (`Date`), а не локальное строковое представление.

## Область изменений

План ориентирован на следующие зоны проекта:

- `SwiftUI-SotkaApp/Services/DailyActivitiesService.swift`
- `SwiftUI-SotkaApp/Libraries/SWUtils/Sources/SWUtils/DateFormatter/DateFormatterService.swift`
- `SwiftUI-SotkaApp/Libraries/SWNetwork/Sources/SWNetwork/Public/JSONDecoder+.swift`
- `SwiftUI-SotkaApp/Services/ProgressSyncService.swift`
- `SwiftUI-SotkaApp/Services/CustomExercisesService.swift`
- `SwiftUI-SotkaApp/Services/StatusManager.swift`
- связанные unit-тесты в `SwiftUI-SotkaAppTests/`

## Приоритет 1. Минимально рискованный hotfix

### Подэтап 1.0. Исправить критерий выбора предыдущей сохраненной тренировки

#### Цель подэтапа

Скорректировать уже внедренную функциональность подстановки предыдущей сохраненной тренировки в новую тренировку так, чтобы она соответствовала продуктовой логике старого приложения и не зависела от `modifyDate`.

#### Что именно нужно исправить

- Исправить выбор "предыдущей тренировки" в логике, добавленной для подстановки данных прошлой тренировки в новую.
- Вместо выбора по самому новому `modifyDate` использовать выбор по максимальному `day` среди тренировок, удовлетворяющих бизнес-условиям.
- Ограничить поиск только тренировками с `day < currentDay`, чтобы выбирать именно предыдущую тренировку, а не просто последнюю измененную запись.
- Сохранить исключение turbo-тренировок.

#### Почему это должно идти первым

- Это уже затрагивает релизную пользовательскую функциональность подстановки прошлой тренировки.
- Сейчас поведение может быть неверным даже без дальнейших изменений sync-слоя.
- Это не полный timezone-fix, но это самый безопасный и самый точечный шаг перед остальными исправлениями.

#### Зоны изменений

- `SwiftUI-SotkaApp/Services/DailyActivitiesService.swift`
- `SwiftUI-SotkaApp/Screens/WorkoutPreview/WorkoutPreviewViewModel.swift`
- `SwiftUI-SotkaApp/Services/StatusManager.swift`
- тесты в `SwiftUI-SotkaAppTests/DailyActivitiesTests/`
- тесты в `SwiftUI-SotkaAppTests/WorkoutPreviewViewModelTests/`
- тесты в `SwiftUI-SotkaAppTests/StatusManagerTests/`

#### TDD-план

##### Red

- Добавить или скорректировать тесты на выбор предыдущей тренировки так, чтобы они фиксировали правильное поведение:
  - выбирается тренировка только с `day < currentDay`;
  - среди подходящих записей выбирается максимальный `day`, а не максимальный `modifyDate`;
  - turbo-тренировки не участвуют;
  - записи с `shouldDelete == true` не участвуют;
  - записи другого пользователя не участвуют.
- Переписать тесты, которые сейчас закрепляют неверное ожидание выбора по `modifyDate`.
- Добавить пользовательские сценарии для двух потребителей этой логики:
  - `WorkoutPreviewViewModel` подставляет данные именно из предыдущей по номеру дня тренировки;
  - `StatusManager` для watch-запроса использует ту же day-based логику.

##### Green

- Минимально изменить контракт или реализацию поиска предыдущей тренировки.
- Предпочтительный вариант: явно передавать `currentDay` в метод поиска и выбирать максимум по `day` среди записей `day < currentDay`.
- Не менять sync/LWW-логику на этом подэтапе.

##### Refactor

- Переименовать метод/комментарии так, чтобы из API было ясно: ищется именно предыдущая сохраненная нетурбо-тренировка, а не просто "последняя измененная".
- Удалить из тестов и комментариев предположение, что корректность определяется по `modifyDate`.

### Цель этапа

Убрать пользовательский эффект, при котором приложение выбирает "не ту" предыдущую тренировку из-за смещенного `modifyDate`.

### Что исправляем

- Перевести выбор последней пройденной нетурбо-тренировки с сортировки по `modifyDate` на выбор по максимальному `day`.
- Сохранить текущую sync-логику без расширения области изменений.

### Почему это первый шаг

- Поведение соответствует старому iOS-приложению.
- Изменение локальное и не требует миграции схемы.
- Исправляет наиболее заметный пользовательский сценарий с минимальным риском для релизной базы.
- Также исправляет уже внедренную функциональность подстановки предыдущей сохраненной тренировки, если она выбрана по неверному критерию.

### TDD-план

#### Red

- Добавить тесты на `DailyActivitiesService.getLastPassedNonTurboWorkoutActivity(context:)`, которые фиксируют ожидаемое поведение по номеру дня:
  - метод должен учитывать номер текущего дня и искать только среди `day < currentDay`;
  - возвращается запись с максимальным `day`, а не с самым новым `modifyDate`;
  - turbo-тренировки не участвуют;
  - записи с `shouldDelete == true` не участвуют;
  - учитывается только текущий пользователь.
- Зафиксировать текущий неверный сценарий: старый день с более поздним `modifyDate` не должен побеждать более новый день программы.
- Зафиксировать текущий неверный сценарий: запись с `day > currentDay` не должна использоваться как "предыдущая" тренировка.

#### Green

- Минимально изменить реализацию поиска последней тренировки.
- Предпочтительный вариант: фильтровать подходящие записи по правилу `day < currentDay` и выбирать максимум по `day`.
- Не менять остальную conflict-resolution логику на этом этапе.

#### Refactor

- Упростить название/комментарии метода так, чтобы поведение было явно day-based.
- При необходимости обновить существующие тесты, в которых ожидание завязано на `modifyDate`.
- Убедиться, что `WorkoutPreviewViewModel` и `StatusManager` используют одну и ту же исправленную логику выбора предыдущей тренировки.

### Критерии завершения hotfix

- Все тесты на выбор предыдущей тренировки зеленые.
- Логика подстановки предыдущей тренировки корректно работает и для iPhone, и для Apple Watch.
- Новое поведение согласовано с логикой старого приложения.
- Никакие другие sync-сценарии не затронуты.

## Приоритет 2. Полный client-only fix

### Цель этапа

Унифицировать работу клиента с датами без серверных изменений и убрать источник timezone-сдвига на клиенте.

### Подэтап 2.1. Нормализация отправки дат на сервер

#### Что исправляем

- Убрать формирование строк вида "локальное время + `Z`".
- Все исходящие даты отправлять либо в UTC, либо с корректным offset, но всегда консистентно.

#### Зоны изменений

- `DateFormatterService.stringFromFullDate(...)`
- формирование запросов для:
  - `DayActivity` / `ActivitySnapshot`
  - `CustomExercise` / `ExerciseSnapshot`
  - `ProgressSyncService`
  - `StatusManager.startNewRun(appDate:)`

#### TDD-план

##### Red

- Добавить unit-тесты на `DateFormatterService`:
  - дата в не-UTC timezone сериализуется в корректную UTC-строку;
  - при `iso: true` результат не должен содержать ложное UTC-обозначение для локального времени;
  - одинаковый `Date` дает одинаковый wire-format независимо от текущей device timezone.
- Добавить тесты на request-building для активностей, прогресса, кастомных упражнений и старта программы.

##### Green

- Исправить форматтер или выделить отдельный API-specific formatter для wire dates.
- Перевести все sync-запросы на него.

##### Refactor

- Явно разделить display formatting и network formatting.
- Убрать двусмысленные параметры и названия, если они мешают безопасному использованию.

### Подэтап 2.2. Нормализация чтения дат с сервера

#### Что исправляем

- Убрать зависимость fallback-парсинга server datetime без timezone от локальной timezone устройства.
- Зафиксировать единое правило интерпретации строк без offset.

#### TDD-план

##### Red

- Добавить тесты на `JSONDecoder.DateDecodingStrategy.flexibleDateDecoding`:
  - ISO8601 с `Z` корректно парсится;
  - ISO8601 с offset корректно парсится;
  - строка без timezone парсится детерминированно и одинаково на устройствах с разными timezone;
  - short date `yyyy-MM-dd` остается стабильной.

##### Green

- Исправить fallback parser для server datetime без timezone.
- Принять одно документированное правило, например: timezone-less server datetime трактуется как server-local convention, но после парсинга сравнивается только как `Date`.

##### Refactor

- Вынести parsing policy в отдельное место или вспомогательный тип, если это улучшит читаемость.
- Обновить комментарии к decoder strategy.

### Подэтап 2.3. Выравнивание sync и LWW-поведения

#### Что исправляем

- Пересмотреть места, где клиент напрямую доверяет server `modifyDate/createDate` в conflict resolution.
- Сохранить текущий принцип offline-first, но убрать ложные выигрыши из-за timezone skew.

#### Зоны изменений

- `DailyActivitiesService`
- `ProgressSyncService`
- `CustomExercisesService`
- `StatusManager`

#### TDD-план

##### Red

- Добавить интеграционно-подобные unit-тесты на сценарии sync:
  - локальная запись не проигрывает серверной только из-за смещенной timezone-строки;
  - после успешной синхронизации серверный ответ не ломает локальный порядок данных;
  - start date не создает ложный конфликт при одинаковом календарном дне.
- Для `DailyActivitiesService` отдельно покрыть сценарий `applySyncEvents` и `downloadServerActivities`.
- Для `ProgressSyncService` и `CustomExercisesService` покрыть сравнение локальной и серверной версий при одинаковых данных и при реальном конфликте.

##### Green

- Внести минимальные изменения в LWW-сравнение только после появления тестов.
- Там, где безопасно, сравнивать только нормализованные `Date`.
- Не вводить day-based логику в местах, где нужен именно временной порядок обновлений.

##### Refactor

- Унифицировать общие правила сравнения дат в sync-слое.
- При необходимости вынести comparison policy в отдельный helper/service, чтобы одна и та же логика не дублировалась в трех сервисах.

### Подэтап 2.4. Осторожная работа с уже существующими локальными данными

#### Что делать

- Не начинать с массовой миграции данных.
- Сначала выкатить исправление будущих записей и убедиться, что новые данные больше не искажаются.
- После этого отдельно оценить, нужен ли узкий data-repair для уже испорченных локальных timestamp.

#### TDD-план

##### Red

- Если будет принято решение о repair-логике, сначала добавить тесты на точечные случаи поврежденных данных.

##### Green

- Реализовать только узкий и идемпотентный repair, если он действительно нужен.

##### Refactor

- Документировать repair-механику и версионирование флага миграции.

### Критерии завершения полного client-only fix

- Исходящие даты сериализуются консистентно и детерминированно.
- Входящие даты парсятся одинаково независимо от timezone устройства.
- Sync-сценарии не зависят от ложного timezone-смещения.
- Поведение покрыто автоматическими тестами, а не только ручной проверкой.

## Приоритет 3. Обязательные тесты

### 1. Тесты на hotfix логики предыдущей тренировки

- `DailyActivitiesServiceTests`:
  - учитывается только `day < currentDay`;
  - выбирается максимальный `day`, а не максимальный `modifyDate`;
  - turbo исключается;
  - deleted исключается;
  - данные другого пользователя исключаются.
- `WorkoutPreviewViewModelTests`:
  - в новую тренировку подставляются данные именно из предыдущей по номеру дня тренировки.
- `StatusManagerTests`:
  - watch-ответ для новой тренировки использует ту же day-based логику выбора предыдущей тренировки.

### 2. Тесты на network date formatting

- `DateFormatterServiceTests`:
  - UTC serialization;
  - serialization для non-UTC локали;
  - отсутствие ложного `Z`;
  - стабильность результата при смене timezone.

### 3. Тесты на request mapping

- `DayActivity` / `ActivitySnapshot`:
  - `createDate` и `modifyDate` в request формируются в корректном wire-format.
- `ExerciseSnapshot` / `CustomExerciseRequest`:
  - даты упражнений уходят в том же формате.
- `ProgressSyncService`:
  - `modifyDate` в запросе прогресса формируется корректно.
- `StatusManager`:
  - дата старта отправляется корректно и детерминированно.

### 4. Тесты на date decoding

- `JSONDecoder+Tests`:
  - ISO8601 с `Z`;
  - ISO8601 с fractional seconds;
  - ISO8601 с explicit offset;
  - timezone-less datetime;
  - short date.

### 5. Тесты на sync conflict resolution

- `DailyActivitiesService`:
  - локальная запись с несинхронизированными изменениями не перетирается сервером;
  - синхронизированная запись корректно обновляется только при реально более новой серверной версии;
  - одинаковые данные не вызывают лишнего обновления.
- `ProgressSyncService`:
  - корректное сравнение `lastModified` и server dates;
  - отсутствие ложного конфликта при timezone skew.
- `CustomExercisesService`:
  - те же сценарии для упражнений.

### 6. Тесты на start date conflict handling

- `StatusManagerTests`:
  - одинаковый календарный день между app/site не создает conflict screen;
  - действительно разные дни создают conflict screen;
  - старт новой программы отправляет корректную дату.

## Рекомендуемый порядок реализации

### Шаг 1. Hotfix

- Написать тесты на выбор предыдущей тренировки и на корректный учет `currentDay`.
- Внести минимальное исправление.
- Запустить целевые тесты.

### Шаг 2. Безопасная починка исходящих дат

- Написать тесты на форматирование и request mapping.
- Исправить network date formatting.
- Прогнать целевые тесты sync-слоя.

### Шаг 3. Починка входящего парсинга

- Написать decoder tests.
- Исправить fallback date parsing.
- Проверить совместимость с существующими ответами API.

### Шаг 4. Починка sync/LWW

- Написать тесты на конфликтные сценарии.
- Точечно обновить `DailyActivitiesService`, `ProgressSyncService`, `CustomExercisesService`, `StatusManager`.
- Прогнать расширенный набор тестов.

### Шаг 5. Решение по repair уже сохраненных данных

- Сначала собрать факты после основных фиксов.
- Только потом решать, нужен ли отдельный repair step.

---

## Что не делать в первой итерации

- Не менять сервер.
- Не делать широкую миграцию локальных данных без тестов и подтвержденной необходимости.
- Не переводить всю sync-логику на сортировку по `day`.
- Не смешивать hotfix с полным рефакторингом date infrastructure в одном PR.

## Definition of Done

- Созданы и зафиксированы failing tests для каждого подэтапа до изменения кода.
- Каждый подэтап доведен до зеленого состояния отдельным минимальным изменением.
- После Green выполнен небольшой Refactor без смены поведения.
- Все новые правила работы с датами задокументированы рядом с тестами и кодом.
- Для hotfix и полного fix можно сделать отдельные PR/релизные поставки.
