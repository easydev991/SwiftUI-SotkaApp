# Работа с документацией проекта

## Когда применять

- При начале работы с любым функционалом проекта - обязательно изучить соответствующую документацию
- При изменении или доработке существующего функционала - обновить документацию
- При добавлении новых экранов или функций - создать или обновить документацию
- При рефакторинге - проверить актуальность документации и обновить примеры
- При обнаружении несоответствий между документацией и кодом - исправить документацию
- При создании новых документационных файлов в папке docs/

## Расположение документации

### Основная документация

- **Папка `docs/`** - основная папка с детальной документацией проекта
- **README.md** - основная информация о проекте, краткое описание функционала и ссылки на детальную документацию

### Структура документации в `docs/`

- `feature-map.md` - карта экранов и функционала
- `data-migration.md` - руководство по миграции данных SwiftData
- `setup-guide.md` - инструкция по установке и настройке
- `deployment.md` - инструкции по публикации приложения
- `infoposts.md` - документация по экрану инфопостов
- `custom-exercises.md` - документация по экрану пользовательских упражнений
- `daily-activities.md` - документация по дневнику тренировок
- `progress.md` - документация по экрану прогресса
- `sync-journal.md` - документация по журналу синхронизаций
- `testing-mocks.md` - документация по мокам для тестирования
- `workout-preview-screen.md` - документация по экрану превью тренировки
- `workout-screen.md` - документация по экрану выполнения тренировки
- `workout-timer-screen.md` - документация по экрану таймера тренировки

## Обязательные правила работы с документацией

### При изменении и доработке функционала

1. **Изучение документации - обязательно**
   - Перед началом работы с любым функционалом **обязательно изучи** соответствующую документацию в папке `docs/`
   - Сверяйся с `feature-map.md` для понимания текущего состояния экранов и функций
   - Изучай специфичную документацию (например, `infoposts.md`, `custom-exercises.md`) перед работой с соответствующим функционалом

2. **Актуализация документации - обязательно**
   - При изменении функционала **обязательно обновляй** соответствующую документацию
   - Если добавляешь новый экран или функцию - обнови `feature-map.md`
   - Если изменяешь логику работы - обнови соответствующую специфичную документацию
   - Если изменяешь структуру данных - обнови `data-migration.md` при необходимости

3. **Согласование изменений**
   - Перед внесением значительных изменений в основную документацию (README.md, feature-map.md) **запрашивай согласование**
   - Мелкие актуализации (например, обновление статуса реализации в feature-map.md) можно делать без согласования

### Принципы качества документации

1. **Актуальность**
   - Документация должна **соответствовать текущему состоянию** кода и функционала
   - При обнаружении несоответствий между документацией и кодом - **немедленно обновляй** документацию
   - Регулярно проверяй актуальность документации при работе с кодом

2. **Лаконичность**
   - Документация должна быть **краткой и по делу**
   - Избегай избыточных описаний и повторений
   - Используй структурированный формат (заголовки, списки, таблицы)
   - Удаляй устаревшую информацию при обновлении

3. **Понятность**
   - Документация должна быть **понятной** для разработчиков, которые работают с проектом
   - Используй простой и ясный язык
   - Добавляй примеры кода там, где это необходимо
   - Описывай не только "что", но и "почему" в сложных случаях

## Правила для Cursor

### При начале работы с функционалом

1. **Прочитай** соответствующую документацию в `docs/`
2. **Изучи** `feature-map.md` для понимания текущего состояния
3. **Сверься** с README.md для общего контекста проекта

### При изменении функционала

1. **Обнови** соответствующие разделы документации
2. **Отметь** изменения в `feature-map.md` (статус реализации, новые функции)
3. **Проверь** актуальность связанной документации

### При создании нового функционала

1. **Создай** или обнови соответствующую документацию в `docs/`
2. **Добавь** описание в `feature-map.md`
3. **Обнови** README.md при необходимости

### При рефакторинге

1. **Проверь** актуальность документации после рефакторинга
2. **Обнови** примеры кода в документации, если они изменились
3. **Удаляй** устаревшую информацию

## Контроль качества документации

### Регулярные проверки

- При каждом изменении функционала проверяй соответствие документации
- При обнаружении несоответствий исправляй их сразу
- При работе с новым функционалом начинай с изучения документации

### Признаки качественной документации

- ✅ Описание соответствует текущему коду
- ✅ Структура понятна и логична
- ✅ Примеры кода актуальны
- ✅ Нет устаревшей информации
- ✅ Документация помогает понять функционал

## Заключение

Документация в папке `docs/` является важной частью проекта и должна поддерживаться в актуальном состоянии. При любой работе с функционалом необходимо изучать и актуализировать документацию, чтобы она была актуальной, лаконичной и понятной для всех разработчиков проекта.
